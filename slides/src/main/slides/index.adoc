=== The pipeline

Every instruction travels through stages:

[%autowidth]
|===
| Stage | What happens

| Fetch | Load instruction from memory (or cache)
| Decode | Figure out what the instruction means
| Execute | Run it on the ALU / load-store unit
| Write-back | Store the result
|===

A modern CPU has *many* of these stages running *in parallel* — this is called *pipelining*

=== The problem with waiting

Some instructions take longer than others

[source,java]
----
x = 1;            // store — fast
y = arr[i];       // load from RAM — ~200 cycles
z = 2;            // fast, but independent of y
----

Sitting idle wastes the pipeline — so the CPU *reorders* independent instructions to keep every stage busy

=== Out-of-order execution

The CPU maintains a *reorder buffer*: a queue of in-flight instructions

* Instructions are *issued* in program order
* Instructions are *executed* as soon as their inputs are ready — regardless of order
* Results are *retired* (committed) in program order — so *single-threaded* code sees no difference

=== Out-of-order execution — how it works

three instructions, one very slow:

[source,java]
----
x = 1;            // store — fast
y = arr[i];       // load from RAM — ~200 cycles
z = 2;            // fast, but independent of y
----

=== !

[%autowidth]
|===
| Clock | Fetch | Decode | Execute | Commit (in order)

| 1     | `x = 1`      |              |                            |
| 2     | `y = arr[i]` | `x = 1`      |                            |
| 3     | `z = 2`      | `y = arr[i]` | `x = 1` ✓                  |
| 4     |              | `z = 2`      | `y = arr[i]` ⏳            | `x = 1`
| 5     |              |              | `z = 2` ✓ *out of order!*  | waiting for `y`...
| 6–203 |              |              | `y = arr[i]` ⏳ ...        | waiting for `y`...
| 204   |              |              | `y = arr[i]` ✓             | `y`, then `z`
|===


=== pipeline hazards

a *pipeline hazard* is any condition that prevents the next instruction from executing in the next clock cycle

* *structural* — resource conflict: two instructions need the same hardware at the same time
* *data* — dependency: an instruction needs a result not yet produced by an earlier instruction
* *control* — branching: the CPU doesn't know which instruction comes next

we solve it by a *pipeline bubble* +
it is an empty slot propagating through the pipeline +
— wasted cycles where no useful work is done

=== cache coherence

[quote]
In a cache coherent system, if multiple clients have a cached copy of the same region of a shared memory resource, all copies are the same. Without cache coherence, a change made to the region by one client may not be seen by others, and errors can result when the data used by different clients is mismatched


=== cache coherence protocols

MESI assigns every cache line one of four states:

[%autowidth]
|===
| State | Meaning

| *Modified*  | only copy; dirty — not yet written back to memory
| *Exclusive* | only copy; clean — matches memory
| *Shared*    | multiple caches hold a clean copy
| *Invalid*   | stale or absent — must fetch before use
|===

every read or write transitions the **cache line** through these states +
and notifies other caches — this is how coherence is maintained

=== snooping

every cache controller *listens* (snoops) to all transactions on the shared bus or interconnect

when it sees a transaction that affects a **cache line** it holds, it reacts:

[%autowidth]
|===
| Event snooped | Line state | Action

| another CPU reads our Modified line
| M → S
| write-back to memory, share the line

| another CPU writes (invalidates) a line we hold
| S / E → I
| mark our copy Invalid immediately

| another CPU reads a line we hold Exclusive
| E → S
| silently downgrade — no write-back needed
|===

no central coordinator — coherence *emerges* from each cache reacting to bus events

== hardware

memory fences and memory models

* memory fence (barrier): a CPU/compiler instruction enforcing ordering constraints on memory operations
* two layers: compiler barrier (prevents reordering in generated code) + CPU barrier (prevents reordering in the store buffer / reorder buffer)
* four types: LoadLoad, LoadStore, StoreStore, StoreLoad — each prevents one reordering pair
* x86 is TSO (Total Store Order): only StoreLoad reordering possible; barriers often implicit
* ARM is weakly ordered: nearly all reorderings allowed; barriers must be explicit

=== memory models

are formal mathematical "contracts" +
between hardware, compilers and developers

when memory writes are visible to other threads 

WARNING: great oversimplification of what memory models are

=== synchronizing actions

operations that establish *happens-before* relationships between threads

[%autowidth]
|===
| Action | Example

| volatile read / write | `volatile int x`
| monitor acquire / release | `synchronized` block
| thread start | `thread.start()`
| thread join | `thread.join()`
|===

if thread A performs a synchronizing action and thread B observes it, +
*all writes by A before that action are guaranteed visible to B*

whether a hardware fence is emitted depends on the architecture — +
the model is hardware-independent

=== testing our expectations

adding a synchronization action causes the compiler and CPU to insert the appropriate barriers

[source,java]
----
class A{
  int x;
  volatile int y;

  void write(){
    x=1;
    y=1;
  }

  void read(Result result){
    result.y=y;
    result.x=x;
  }
}
----

* write to synchronization variable: StoreStore barrier before + StoreLoad barrier after
* read from synchronization variable: LoadLoad + LoadStore barrier after
* `x=1` is committed to the cache hierarchy before `y=1` is made visible → `x=0, y=1` outcome is now *impossible*

=== SC-DRF

there is a middle ground, an agreement in community

“data-race-free sequential-consistency”

[quote]
Instead of guaranteeing sequential consistency, as we mentioned earlier, today’s processors guarantee a property called “data-race-free sequential-consistency”, or DRF-SC (sometimes also written SC-DRF). A system guaranteeing DRF-SC must define specific instructions called synchronizing instructions, which provide a way to coordinate different processors (equivalently, threads). Programs use those instructions to create a “happens before” relationship between code running on one processor and code running on another.

=== challenges

* progress guarantees
** *deadlock*: threads wait for each other's lock — no thread ever makes progress
** *livelock*: threads keep reacting to each other — activity without progress
** *starvation*: a thread is perpetually denied access while others keep taking priority
** *lock-free*: at least one thread always makes progress
** *wait-free*: every thread makes progress in a bounded number of steps — strongest guarantee, hardest to implement
* *latency*: caused by contention on locks, can be a killer for systems performance

=== immutability changes everything

* immutable data needs no synchronization for reads — once published, any number of threads can read concurrently without barriers
* only synchronization needed: *safe publication* of the reference — the object must be fully initialized before any other thread can observe it
* reads scale linearly with core count — no contention, no cache-line bouncing
* functional languages and immutable-by-default designs compose naturally under concurrency

=== communicating sequential processes

Tony Hoare, 1978 — _Communicating Sequential Processes_

[quote]
Do not communicate by sharing memory; instead, share memory by communicating.

* concurrent *processes* run in isolation — no shared state
* coordination happens exclusively through *channels*: typed conduits for passing messages
* sender blocks until receiver is ready — a *synchronous rendezvous* (or buffered up to a capacity)
* the channel *is* the synchronization — no locks, no volatiles needed

=== pros and cons

[%autowidth]
|===
| ✓ Pros | ✗ Cons

| no shared mutable state → no data races by construction
| deadlock still possible: two processes waiting on each other's channel

| deadlock detection is tractable — Hoare's CSP algebra; model checkers (FDR, SPIN)
| buffered channels re-introduce coordination complexity

| composable: pipelines, fan-out, fan-in are natural idioms
| not ideal for fine-grained shared data (counters, caches) — other primitives still needed

| mental model maps naturally to real-world workflows
| channel-heavy code is hard to trace — "goroutine leak" is a real production hazard

|===

=== actor model

Carl Hewitt, 1973 — _A Universal Modular ACTOR Formalism for Artificial Intelligence_

* an *actor* is the fundamental unit of computation: it has private state, a *mailbox*, and a behavior
* actors communicate exclusively by sending *asynchronous messages* — no shared memory, no blocking
* on receiving a message an actor can: send messages to other actors, create new actors, change its own behavior
* the runtime routes messages; actors run wherever the scheduler decides — location is transparent

=== pros and cons

[%autowidth]
|===
| ✓ Pros | ✗ Cons

| no shared state → no data races
| async messaging makes reasoning about order and timing harder

| natural fault isolation — one actor crashing doesn't affect others
| mailbox overflow if consumers are slower than producers

| scales to distributed systems; location transparency is built in
| debugging message flows across many actors is difficult

| "let it crash" + supervision trees simplify error recovery
| not a silver bullet for coordinated transactions — eventual consistency only

|===

=== Actors vs CSP

[%autowidth]
|===
| | Actor Model | CSP

| *communication*
| async — message queued in recipient's mailbox
| sync (or buffered) — rendezvous on a shared channel

| *identity*
| addressed by actor reference
| addressed by channel reference

| *failure handling*
| built-in supervision / "let it crash"
| handled outside the model

| *location*
| location-transparent; actors can be remote
| channels are typically local / in-process

|===

=== formal verification

verify correctness of a *design* before writing a line of code

tools: *TLA+* (Leslie Lamport, used by AWS at scale), *SPIN*, *Alloy* +
exhaustively explores *all* possible execution states — no interleaving is missed

* *✓* finds subtle bugs that no amount of runtime testing can catch
* *✓* proves correctness — not just "no bug found so far"
* *✗* verifies the *spec*, not the actual implementation code
* *✗* state explosion — tractable only for small, well-bounded models
* *✗* steep learning curve: temporal logic, TLA+ / Promela spec languages

=== stress testing

run the concurrent scenario many times and *hope* it triggers a failure

* *✓* works on real production code — zero model-building effort
* *✓* simple annotation-based API, familiar to Java developers
* *✗* probabilistic — needs thousands of iterations to build confidence
